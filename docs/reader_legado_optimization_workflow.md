# Reader x Legado 工作流二：主動維護（迭代優化）

## 目的

- 這份工作流用在沒有明確 bug，但想主動整理、穩定化、優化 `reader` 現有功能時。
- 目標是借鏡 `legado` 的邊界切法與流程安排，改善 `reader` 的現有模組，而不是新增產品功能。

## 怎麼使用 `reader_legado_index.md`

1. 先打開 [reader_legado_index.md](/home/benny/projects/reader/docs/reader_legado_index.md)
2. 從「模組清單」選一個這次要動的功能區
3. 先讀該模組的 2-4 行摘要，確認這次範圍真的只落在這一塊
4. 再切去 `docs/reader_legado/` 裡對應的模組文檔
5. 在模組文檔裡優先看：
   - `Reader 現況`
   - `Reader 上下游依賴`
   - `可借鏡的方向`

## 工作流

### 1. 選定功能區

- 從地圖選一個功能，明確說這次只動這一塊

### 2. 現況盤點

- 把這個功能區的上游（依賴）和下游（被誰用）列出來
- 確認目前的責任切分是否清晰

### 3. 對照 legado

- 看地圖對應的 `legado` 資料夾
- 問：它的流程切法有什麼可以借鏡的？
- 列出 2-3 個具體的優化方向

### 4. 規劃優化

- 每次只動一個層（UI / 服務 / 核心）
- 先穩定化，後重構
- 不在這次引入新功能

### 5. 驗收邊界

- 確認上下游行為沒有改變
- 更新地圖的「現況」描述（如果結構有變）

## 使用原則

- 一次只選一個模組，不跨多塊一起整理
- 先看上下游依賴，再決定從哪一層開始
- `legado` 只作為流程與責任切分的參考
- 在實作前，先提交本次維護摘要與預計修改範圍，等待確認
- 摘要裡要明確說明改變之前和改變之後會是什麼樣子，包含使用者可見行為與內部責任切分
- 經確認後再開始修改
- 修改完成並通過驗收後，要提交本次變更的 commit
- 如果結構改了，最後要同步更新對應模組文檔
