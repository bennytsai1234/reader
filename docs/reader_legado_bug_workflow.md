# Reader x Legado Bug 工作流程

## 目的

- 用於明確 bug、錯誤症狀、crash、失敗測試、回歸或不穩定行為。
- 一般 bug 修復不要重新執行 Codebase Atlas；先用 [reader_legado_index.md](reader_legado_index.md) 定位，再進入模組文檔與程式碼。
- `legado` 可作為責任邊界、流程順序、失敗處理與 parity 行為參考，但 bug owner 仍以 `reader` 的現有程式碼與測試為準。

## 工作流程

1. 保留使用者原始症狀、錯誤訊息、失敗測試或重現條件。
2. 打開 [reader_legado_index.md](reader_legado_index.md)，用症狀選出最可能的主模組。
3. 閱讀該模組的「目標專案目前狀態」、「目標專案上下游」、「目標專案變更入口」、「目標專案變更路線」、「已知風險」。
4. 進入實際程式碼、測試、log 或重現路徑，確認問題是在主模組內發生，還是由上游輸入造成。
5. 若症狀跨模組，依資料、狀態或行為第一次變錯的位置選主模組；其他模組只當邊界模組，改動保持最小。
6. 只查該模組文檔列出的 `legado` 對應區域，借鏡責任切分、防呆點與 parity 差異，不查 unrelated feature。
7. 修復集中在主模組；跨模組或邊界改動必須有明確原因。
8. 驗證 bug 已修復，且上下游未回歸。
9. 若結構、責任、API 或主要流程改變，更新受影響的 Atlas 模組文檔。
10. 依交付政策完成。

## 修改前／修改後閘門

當此工作流程被呼叫時，不要先改 code。先用 Atlas 與程式碼檢查完成分析，並提供給使用者確認。

分析上半部可以包含工程細節，例如主模組、根因、受影響檔案、風險與檢查。最後部分必須只保留白話的修改前／修改後摘要：

- **修改前**：說明目前行為如何運作，以及錯在哪裡。
- **修改後**：說明修復後會產生什麼行為。

等待使用者明確確認後，才可以編輯檔案修復 bug。

## 交付政策

提交並推送。完成驗證後建立聚焦 commit，確認目標 remote/branch 後推送；若 remote 或 branch 不清楚，推送前必須先詢問。
