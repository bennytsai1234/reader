# Reader x Legado 工作流一：遇到 Bug

## 目的

- 這份工作流用在 `reader` 已經出現明確 bug、錯誤症狀或不穩定行為時。
- 目標不是擴需求，而是先把問題定位在正確邊界，再去參考 `legado` 的對應策略做修復。

## 怎麼使用 `reader_legado_index.md`

1. 先打開 [reader_legado_index.md](/home/benny/projects/reader/docs/reader_legado_index.md)
2. 用症狀去對照「模組清單」和「每個模組的 2-4 行摘要」
3. 找到最接近的模組後，再切去 `docs/reader_legado/` 裡對應的模組文檔
4. 在模組文檔裡先看：
   - `Reader 現況`
   - `Reader 上下游依賴`
   - `Legado 對照`
5. 之後才進入程式碼查找與修復

## 工作流

### 1. 定位

- 描述症狀
- 對照地圖找到「哪個功能區」
- 確認是 UI 層 / 服務層 / 核心層 的哪一層

### 2. 往上游走

- 查這個功能區的直接依賴，例如 `core/services`、`core/engine`
- 確認資料是從哪裡來的、狀態在哪裡被改

### 3. 比對 legado 策略

- 只看地圖裡對應的 `legado` 資料夾
- 問：它在這個邊界上是怎麼切責任的？
- 不是問：它有沒有更多功能

### 4. 修復邊界

- 修改集中在問題所在層，不跨層
- 確認修改不破壞上下游

## 使用原則

- 先用索引定位模組，再進入模組文檔，不要直接從大範圍亂找
- 先看 `Reader 上下游依賴`，再看程式碼
- `legado` 只用來借鏡責任切法與穩定化方式
- 在實作前，先提交本次維護摘要與預計修改範圍，等待確認
- 摘要裡要明確說明改變之前和改變之後會是什麼樣子，包含使用者可見行為與內部責任切分
- 經確認後再開始修改
- 修改完成並通過驗收後，要提交本次變更的 commit
- 不因為 bug 修復而順手擴功能
